# Human Feedbacks

## Canvas Pane Layout

### The scroll behavior in canvas-pane is not correct.
The "relative canvas-pane__scroll" should enforce internal scroll rather than the whole web page scroll when the content overflows. Currently, when the content in canvas-pane overflows, the whole web page scrolls instead of just the canvas-pane. I expect only the canvas-pane to scroll internally while the rest of the page remains static. The same UX should be applied to the message-list in context-pane as well.

### Enable toggle canvas-pane on-demand
When the canvas-pane is visible, there should be a button to hide the canvas-pane and expand the context-pane to full width. Conversely, when the canvas-pane is hidden, there should be a button to show the canvas-pane again. This toggle button should be placed near the resizable divider between context-pane and canvas-pane for easy access. The button icon should indicate the current state (show/hide) of the canvas-pane.
Should also enforce a min-width for both context-pane and canvas-pane to avoid layout breaking when resizing the divider.

Notice: should double check and refine the canvas-pane-visibility-rules.md accordingly after implementing this feature to ensure consistency between documentation and actual behavior.

## Context Pane Layout

### Message List Layout
The message-list-container -> message-list spacing is not appropriate when the conversation starts. When there are no messages, the spacing of message-list should be same as when there are messages. Currently, there is too many spaces on the left, right and bottom, in order words, the message-list seems to be sticky to the top of the container. I expect the message-list should span all width and height of the container and the spacing should be consistent regardless of whether there are messages or not. And the chat-container should always sticky to the bottom of the container.

When canvas-pane is visible, the message-list-container must maintain a padding horizontally so that the messages do not touch the edge of the container when moving the resizable divider. I expect the message-list-container to have a consistent padding horizontally regardless of the canvas-pane visibility.

### Chat Container Max-Width
Re-adjusting the width of chat-container, currently it's hard-coded as `max-width: 52rem` when (min-width: 768px) which is not good at all. This would results in very narrow chat area on large screens. I suggest using a relative unit such as percentage or vw for max-width so that the chat area can adapt to different screen sizes better. Should also check for any other media queries (if any).

### Panic Button Positioning
Should relocate the panic-button-container to be next to the send button in ChatInput component. This would make more sense from UX perspective since the send button is the primary action button in chat input area. The panic button should use danger color from Claude theme to indicate its critical function.
Also should animate the panic button with a subtle pulse effect to draw user attention without being too obtrusive.

### Assistant Thought
The "thought-toggle" in "assistant-thought" still does not work. Clicking the toggle button does not expand or collapse the assistant-thought component after the final message is displayed. This issue occurs both during streaming and after completion, indicating that the toggle state is not being updated correctly in response to lifecycle events. The root cause appears to be related to how the component manages its expanded/collapsed state and how it interacts with Blazor's rendering lifecycle.
This issue has persisted after many time being feedbacked and should be fixed as soon as possible since it is a critical UX issue. I suppose this is a trivial fix but for some reasons it has not been resolved after so many iterations.

## AG-UI Features

### Humand in the loop AG-UI feature
When I click on approval button "yes" and the "tool-call-name" -> "request_approval" still get displayed as running svg. The tool-result-section also has tool-result-section-pending class which is not expected after approval. I expect the "tool-call-name" to update to "approved" and the pending class to be removed from tool-result-section.

Also there is another empty assistant-thought expanded displayed directly below the above assistant-thought expanded which is unexpected. I expect there is always only one assistant-thought expanded at a time.

### Tool-based UI Rendering AG-UI feature
I sent message "What is the weather in Seattle?" and the WeatherDisplay component is still not rendered in the message-list as expected. I expect after sending the message, the WeatherDisplay component should be rendered in the message-list as part of assistant's response.

### Shared State AG-UI feature
When I sent message "Show me a simple plan", then the "Recipe State Change" is visible in the canvas-pane, however this is not expected because by default this diff view should be hidden. I expect the Recipe State Change view should be hidden by default unless user explicitly clicks on "Show Recipe State Change" button to reveal it. 
I suggest designing a toolbar in canvas-pane to host such buttons for better UX. These buttons would include: diff view, memory inspector, tool execution tracker, checkpoint manager, etc. Also design UXUI for how to display these components in canvas-pane properly and still maintain good focus on main artifact being rendered. I suggest using tab-like interface to switch between different components in canvas-pane.
In addition, ensure when resizing the divider between context-pane and canvas-pane, the content in canvas-pane should adapt accordingly without breaking layout. For example, the "recipe-editor" should reflow its content based on the available width in canvas-pane.
