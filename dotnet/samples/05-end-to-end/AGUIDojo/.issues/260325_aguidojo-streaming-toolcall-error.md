## Summary

Resolved. The required 3-prompt runtime flow now passes on a clean stack (`server:5110`, `client:6002`) instead of failing on the 3rd message with `HTTP 400 invalid_request_error`.

1. Start AGUIDojo (server+client sample) in the usual environment.
2. Send prompt 1: "Which tools are you provided?"

## Actual behavior

- `StreamingError` occurred at the 3rd user message.
- API error: `HTTP 400 (invalid_request_error: ) Parameter: messages.[6].role An assistant message with 'tool_calls' must be followed by tool messages responding to each 'tool_call_id'. The following tool_call_ids did not have response messages: call_mZYyxrYIfDLlLKGGijL7R2J1`
- A secondary variant also appeared when duplicate tool results were replayed: an orphan/duplicate `tool` message error on the next request.
- Rehydrated/persisted history could also leak duplicate tool results and break structured replay semantics.

## Expected behavior

- The runtime should generate a valid assistant+tool call exchange sequence, including the required tool response messages for every `tool_call_id`.
- Replayed/persisted history should not duplicate tool results or leak server-only replay markers into follow-up client-tool requests.

## Root cause

The failure turned out to be a coupled replay problem rather than a single UI-only bug:

1. outbound AG-UI history shaping could replay an invalid sequence on the next `/chat` call
   - the follow-up payload could contain `assistant(tool_calls)` without the matching `tool` response message
   - server-only replay content could also leak back into a client-tool follow-up request
2. persisted conversation history could retain duplicate `FunctionResultContent` entries for the same `tool_call_id`
   - that corrupted replayability and could produce the sibling duplicate/orphan `tool` error

The immediate 400 repro was confirmed against the live `/chat` payload shape. Persistence duplication was a real secondary defect and was fixed alongside the outbound history shaping.

## Resolution

The final fix was split across the outbound AGUI layer and the AGUIDojo server persistence layer:

- `Microsoft.Agents.AI.AGUI/Shared/AGUIChatMessageExtensions.cs`
  - buffer assistant tool-call turns until matching tool results arrive
  - drop incomplete assistant tool-call turns when a later non-tool boundary proves they were abandoned
  - keep standalone tool messages only when they contain meaningful payloads
  - suppress orphaned tool results from abandoned assistant turns
  - exclude server replay markers from outbound AGUI conversion
- `Microsoft.Agents.AI.AGUI/AGUIChatClient.cs`
  - prune prior server-only tool-call history from follow-up client-tool requests
- `AGUIDojoServer/ChatSessions/ChatConversationService.cs`
  - normalize duplicate `FunctionResultContent` entries during persistence/readback
- `AGUIDojoServer/ChatSessions/ChatSessionWorkspaceService.cs`
  - deduplicate duplicate tool-result content during workspace rehydration/derived-state reads
